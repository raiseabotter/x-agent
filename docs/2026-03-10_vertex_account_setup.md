## 4. Mullvad VPN / RemotePC 展開

### 重要な発見: Split Tunneling の制約

**問題**: Mullvad のアプリケーション単位 split tunneling は、NagiとVertexが**同じPlaywright/Chromiumバイナリ**を使うため、プロセスレベルでの分離ができない。

**推奨方式**: 以下のいずれか（優先度順）

| 方式 | 分離度 | 複雑さ | 推奨 |
|------|--------|--------|------|
| **A. 別Windowsユーザーアカウント** | 高 | 中 | 推奨 |
| B. VM (Hyper-V / VirtualBox) | 最高 | 高 | 確実だが重い |
| C. Docker container | 高 | 中 | Linux前提 |
| D. 時間帯分離 (同一ユーザー) | 低 | 低 | 暫定措置 |

### 方式A: 別Windowsユーザーアカウント（推奨）

```
RemotePC (X1Carbon, Tailscale 100.74.211.55)
├── User: USER (既存)
│   └── Nagi x-agent → Mullvad Exit Node A (例: Tokyo)
│       └── C:/Users/USER/x-agent/ (persona=nagi)
│
└── User: VERTEX (新規作成)
    └── Vertex x-agent → Mullvad Exit Node B (例: Chicago)
        └── C:/Users/VERTEX/x-agent/ (persona=vertex)
```

**セットアップ手順**:

1. RemotePC に新規ユーザー作成:
   ```powershell
   # RemotePC 上で実行 (SSH経由 or RDP)
   net user VERTEX <password> /add
   net localgroup Administrators VERTEX /add
   ```

2. Mullvad VPN インストール（まだの場合）:
   ```powershell
   # winget or 公式サイトから
   winget install MullvadVPN
   ```

3. Mullvad split tunneling 設定:
   - **Mullvad自体はマシン全体に1インスタンスしか動かない**
   - 方式A の場合: 各ユーザーセッションで Mullvad CLI で出口ノードを切り替え
   - または WireGuard 手動設定で別トンネルを構築

4. **より確実な方式: WireGuard 手動設定**:
   ```
   # USER (Nagi): wg0 interface → Mullvad Tokyo server
   # VERTEX:      wg1 interface → Mullvad Chicago server
   ```
   - Mullvad のWireGuardキーをアカウントページから取得
   - 各ユーザーの routing table で該当インターフェースを指定

### 方式D: 時間帯分離（暫定措置、Phase A向け）

NagiとVertexのactive_hoursが異なるため、暫定的に同一ユーザーで運用可能:

| 時間帯 (UTC) | Nagi (JST) | Vertex (CT) | 競合 |
|-------------|------------|-------------|------|
| 22:00-05:00 | 07:00-14:00 活発 | 16:00-23:00 夜 | 低 |
| 05:00-13:00 | 14:00-22:00 活発 | 23:00-07:00 停止 | なし |
| 13:00-22:00 | 22:00-07:00 停止 | 07:00-16:00 活発 | なし |

**→ タイムゾーン差により、同時稼働する時間帯はほぼない。Phase A (likes only) では方式Dで十分。**
